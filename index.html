0 핵심 한 문장 (이것만 외워도 됨)

  • 인증 — 비밀번호는 해시로만 저장하고, 로그인하면 토큰으로 신원을 증명한다.
  • OAuth — 비밀번호를 우리가 안 받고 구글·깃허브·애플에 위임한다.
  • 결제 — 카드 정보는 우리가 안 만지고 Stripe가 처리한다.
  • 알림 — 모바일 푸시(FCM) + 이메일(SES) 두 채널.
  • 감사 — 누가 뭘 했는지 중요한 활동을 기록으로 남긴다.

1 보안 (제일 많이 나옴)

Q. 비밀번호 어떻게 저장하나?

원문 저장 안 함. BCrypt 해시로 저장. 일부러 느린 해시라 무차별 대입에 강하고, DB가 털려도 원문 복원 불가.

Q. 토큰이 탈취되면?

실제 요청용 토큰(Access)은 15분만 유효 → 피해 창이 짧다. 갱신용 토큰(Refresh)은 7일이지만 해시로만 저장 → DB가 털려도 재사용 불가.

Q. 탈취 토큰 재사용은?

Refresh 토큰은 한 번 쓰면 폐기하고 새로 발급(회전). 폐기된 토큰이 다시 오면 탈취로 보고 차단.

Q. 토큰 위변조는? (권한 위조)

토큰은 서버 개인키로 서명돼 있어 내용을 바꾸면 검증에서 걸린다. 권한을 admin으로 위조 불가.

Q. 브라우저에서 토큰 탈취(XSS)는?

자바스크립트가 못 읽는 HttpOnly 쿠키에 보관해 막는다.

Q. 무차별 대입 로그인은?

로그인 실패 횟수를 세서 일정 횟수 넘으면 계정 잠금.

Q. 2단계 인증(MFA)은?

비밀번호 + OTP 앱 6자리 코드. 폰 분실 대비 백업 코드도 발급.

Q. 민감정보 암호화는?

MFA 비밀키·소셜 토큰 등은 AES 암호화 저장. 통신은 전부 HTTPS. 비밀번호·토큰은 로그에도 마스킹. 시크릿은 코드에 안 넣고 환경변수/시크릿 매니저로 주입.

막판 2종 세트 — ① 비밀번호 = BCrypt 해시(원문 저장 X) ② 토큰 탈취 = 짧은 수명 + Refresh 해시저장 + 회전/재사용 감지

2 OAuth — "왜 저 세 개만?"

타깃 사용자가 개발자·대학원생·자격증 준비자라 거기 맞는 provider를 골랐다.

  • 깃허브 개발자 거의 다 씀
  • 구글 범용 표준 (한국에서도 보편적)
  • 애플 모바일(iOS) 필수

카카오·네이버는 한국 일반 소비자 대상이라 우선순위 낮음. 우리는 달러 가격 + 다국어의 글로벌 SaaS 포지션. MVP라 핵심 3종으로 좁혀 빠르게 출시, 확장은 쉬움(설정+정보 매핑만 추가) → 일반 사용자 확장 시 카카오 추가 예정.

보안 포인트 (자세히) — 두 축으로 설명하면 된다.
① 토큰이 브라우저에 노출되지 않는다

구글·깃허브가 발급한 access_token은 그 사용자의 프로필·권한에 접근하는 열쇠라, 새면 계정이 털린 것과 같다. 그래서 "인증 코드 → 토큰" 교환은 전부 서버 ↔ 구글 사이에서만 일어나고, 토큰은 서버에만 저장한다. 브라우저에는 절대 안 내려보낸다. 저장 시에도 평문이 아니라 AES로 암호화 → XSS로 브라우저가 뚫리거나 DB 일부가 노출돼도 토큰이 그대로 새지 않는다.

② state로 CSRF(로그인 가로채기) 차단

CSRF란: 공격자가 피해자 브라우저를 속여, 피해자가 모르게 공격자 계정으로 로그인되게 만드는 공격. 성공하면 이후 피해자가 입력한 정보가 공격자 계정에 쌓인다(OAuth에선 "로그인 CSRF").

  • 로그인 시작 시 서버가 추측 불가능한 랜덤값(state)을 만들어 ⓐ HttpOnly 쿠키에 저장 + ⓑ 구글로 보내는 요청에도 함께 실어 보냄
  • 구글이 로그인을 끝내고 사용자를 우리 서버로 돌려보낼 때 그 state를 그대로 다시 줌
  • 서버는 "쿠키의 state == 돌아온 state"인지 비교 → 다르면 내가 시작한 로그인이 아니므로 거부
  • state는 1회용이라 검증 후 즉시 폐기 + 만료도 짧게(재사용·재생 공격 방지). 공격자는 이 값을 미리 알 수 없어 위조 불가

3 알림 (Notification)

Q. 어떻게 보내나?

두 채널 — 모바일 푸시(FCM) + 이메일(AWS SES). 다른 서비스에서 "알림 보내라" 이벤트가 오면 발송.

Q. 사용자가 제어할 수 있나?

알림 끄기/켜기, 방해 금지 시간대, 읽음 처리, 안 읽은 개수 조회 가능. 사용자당 기기 토큰 최대 5개.

Q. 발송이 실패하면?

재시도(횟수·간격 설정). 그래도 안 되면 기록.

Q. 같은 알림이 두 번 가지 않나?

"같은 이벤트 + 같은 채널"은 한 번만 발송(멱등성). 이메일은 사용자당 하루 10건 상한으로 스팸 방지.

4 관리자 (Admin)

Q. 관리자는 뭘 할 수 있나?

사용자 목록 조회·검색, 계정 정지/삭제, 테넌트 관리, 감사 로그 조회, 대시보드 요약 통계, 시스템 설정 변경.

Q. 아무나 못 들어가게 어떻게 막나?

관리자 권한(ROLE_ADMIN) 있는 사람만 접근. 권한은 DB로 관리하고 토큰에 담겨 검증된다.

Q. 관리자 본인 보호는?

관리자가 자기 자신을 정지·삭제하는 건 막아둠(실수 방지).

Q. 정지/삭제하면 바로 막히나?

기존 로그인 세션을 무효화 → 예전 토큰으로 계속 쓰는 걸 차단.

Q. 첫 관리자는 어떻게 만드나?

자동 생성 없이, 가입된 사용자에게 DB 작업으로 권한을 직접 부여. 자동 생성 경로를 안 만든 것 자체가 보안 조치.

5 감사 로그 (Audit) — 보안과 연결

누가 뭘 했는지(로그인, 삭제, 결제, 권한 변경 등) 중요한 활동을 기록으로 남긴다. 다른 서비스에서 일어난 일도 이벤트로 받아 한곳에 모으고, 관리자가 조회. → 사고 시 추적 가능, GDPR 등 규제 대응에도 쓰인다.

6 결제 (Billing) 보안 요약

카드번호·CVC는 우리가 받지도 저장하지도 않음 → PCI 책임을 Stripe로 넘김. Stripe의 결제완료 통지(Webhook)는 서명 검증으로 위조 차단. 같은 통지가 중복으로 와도 이미 처리한 건 무시(멱등성).

7 헷갈리기 쉬운 용어 정리

비밀번호 표현 — "암호화"라고 말하지 말 것

BCrypt는 암호화(encryption)가 아니라 해싱(hashing)이다.

  • 암호화 = 키로 풀면 원문 복구 가능 (양방향)
  • 해싱 = 단방향, 원문 복구 불가

✅ 정확한 답변: "비밀번호는 BCrypt로 해시해서 해시값만 저장합니다. 단방향이라 DB가 털려도 원문 복원이 안 되고, BCrypt는 일부러 느려서 무차별 대입에도 강합니다."

state (OAuth)

소셜 로그인에서 CSRF(위조 요청)를 막는 1회용 검증값. 로그인 시작 시 서버가 랜덤값을 만들어 쿠키에 저장 + 구글에 같이 보냄 → 구글이 콜백에서 그 값을 그대로 돌려줌 → "내가 보낸 값과 같나?" 비교. 다르면 공격자가 끼워넣은 요청이라 거부. → "이 로그인이 진짜 우리 사용자가 시작한 게 맞다"는 도장.

GDPR

EU의 개인정보보호 규정. EU 사용자 정보를 다루는 서비스가 지켜야 하고 위반 시 과징금이 큼. 글로벌 SaaS라 필수.

  • 데이터 내보내기 권리 → data-export API
  • 삭제(잊혀질) 권리 → 계정 삭제 API (30일 유예)

유사 규제: 미국 CCPA, 한국 개인정보보호법.

관리자 시스템 설정 = 뭘 on/off 하나
  • 기능 플래그(feature flag): 코드 재배포 없이 기능을 켜고 끄는 스위치 (예: "AI 카드 자동 생성" 베타 일부만 켜기, 문제 시 즉시 끄기)
  • 요청 제한(rate limit): 분당 API 요청 허용량 같은 운영 수치 조정 (예: 분당 1000건 제한 → 과부하·악용 방지)

8 추가 예상 질문 (⭐ 두 개는 함정 단골)

Q. JWT는 stateless인데 로그아웃은 어떻게 하나?

Access 토큰은 서버가 안 들고 있어서 만료 전 강제 차단이 어렵다. → 로그아웃 시 Refresh 토큰을 삭제해 갱신을 막는다. Access는 15분이라 곧 만료되고, 즉시 차단이 필요하면 세션 무효화로 막는다.

Q. 테넌트(tenant)가 뭐예요?
